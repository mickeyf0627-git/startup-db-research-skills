# startup-db-research: スタートアップDB構築・マッチングDB生成スキル

このスキルは**2つの能力**を持つ。依頼文に応じて使い分ける:

| 能力 | トリガー(主な起動文) | 入力 | 出力 |
|---|---|---|---|
| **能力1: スタートアップDB作成** | 「スタートアップDBを作成して」「○社集めて」 | 社数(任意)・領域(任意) | Excel(①サマリ/②項目定義/③スタートアップDB)+JSONL+中間CSV |
| **能力2: マッチングDB作成** | 「マッチングDBを作成して」+**技術アセットのExcel/CSVの提示** | アセット棚卸しファイル+既存のスタートアップDB | 同じExcelに**④マッチングDB**(キー4列+DB転記4列+本編30列+要約13列=51列)と**⑤アセット情報**(使用アセットの転記)を追加 |

出力Excelのシート構成は常に: **①サマリ/②項目定義/③スタートアップDB/④マッチングDB/⑤アセット情報(④⑤はマッチング実行時のみ)**。テンプレート: [templates/スタートアップDB_テンプレート.xlsx](templates/スタートアップDB_テンプレート.xlsx)

## 事実性の鉄則(商用データベースのため最優先)

このDBは商用利用される。**もっともらしい嘘(ハルシネーション)が1件あるだけでDB全体の信頼が落ちる。** 以下を全フィールドの記入で守ること:

1. **記憶で書かない。** 企業に関する値は、**この実行中にAPI・WebFetchで実際に取得したソース**に書かれていることだけを書く。訓練知識にある「知っている会社」でも、必ず現時点のソースで確認してから書く(買収・上場・ピボットは頻繁に起きる)。
2. **書いた値は出典で指せること。** 各社の `source_urls` のいずれかを開けば、その値が確認できる状態にする。確認できない値は書かない。
3. **不明は null(空欄)。** 空欄は失敗ではない。「埋まっているが間違い」の方がはるかに重大。それらしい値で埋めたくなったら空欄にする。
4. **数字と固有名詞は特に危険。** 調達額・調達日・ラウンド名・投資家名・顧客名は、ソースにその形で書かれているものだけを書く。通貨換算・累計の自力合算をした場合は「(換算)」「(合算)」と明記する。
5. **推定には必ず「(推定)」を付ける。**
6. **ソース間で食い違ったら**: より新しい一次情報(本人発表)を優先し、「(推定)」を付けて両方のURLを `source_urls` に残す。
7. **作ってはいけないもの**: 存在しない製品名、公表されていない顧客・パートナー名、発表されていない資金調達。市場規模・CAGRは**出典レポートが確認できた場合のみ**記入。
8. **収載前の実在チェック**: 未上場か・買収されていないか・事業継続中かを確認(疑わしければ `fit=△` と注記)。
   - **fit判定は定義6基準のみで行う。**「対象領域と関係ない」「クエリの誤ヒット」は△の理由にしない(業界分類で表現する)。
   - `notable_customers_partners` 等に書いた事実の出典URLは、必ず `source_urls` に含める。
9. **納品前セルフチェック**: 完了報告の前に無作為に3〜5社(マッチングDBは3〜5行)を選び全フィールドを出典と突合。不一致があれば全体を再点検し、結果を完了報告に書く。
10. **マッチングDBのスコアは「評価」であり事実ではない**が、その根拠(エージェントの思考根拠)には事実(両者の技術・実績)のみを引用すること。

# 能力1: スタートアップDB作成

## 定義・分類・収集方法(共通ルール)

- **スタートアップの定義(収載6基準)**: ①未上場(買収済み除外) ②設立10年以内(ディープテックは15年) ③外部エクイティ調達実績あり(国内は必須) ④成長志向 ⑤従業員500名未満目安 ⑥独立性。詳細な運用は [references/queries.md](references/queries.md)
- **業界分類(20分類)**: AI・データ基盤/エンタープライズSaaS・業務DX/フィンテック・保険/ヘルスケア・医療機器/バイオ・創薬/モビリティ・自動車/宇宙・航空・防衛/エネルギー・クリーンテック/素材・化学/半導体・エレクトロニクス/ロボティクス・ドローン/製造DX・産業機器/物流・サプライチェーン/農業・食品/建設・不動産・インフラ/小売・EC・消費者サービス/HR・教育/セキュリティ・通信/エンタメ・メディア・ゲーム/その他
- **製造技術分類(8分類・該当時のみ追加タグ)**: AI・データ(シミュレーション/CAE)/センシング/検査・品質/予知保全/ロボティクス・FA/エッジAI/材料・MI/熱・パワエレ
- **社数**: ユーザー指定に従う。未指定時は推定MAX(トライアル時: グローバル約100+国内150〜250 ≈ 300〜350社/実行)を提示して確認。
- **収集**: グローバル=Coresignal API(キーは`APIキー.txt`、ヘッダー`apikey`、ベースURL `https://api.coresignal.com/cdapi/v2/`)。国内=API申請不要ソース(J-Startup/SBIR/NEDO+PR TIMES RSS+KEPPLE・BRIDGE記事)。国内候補は**中間CSV** `output/domestic_candidates_YYYYMMDD.csv` に必ず保存。

## 追記モード(既存DBへの追加収集) — 2回目以降の実行は必ずこれ

プロジェクトに既存のスタートアップDB(`output/startups_master_*.jsonl` または `スタートアップDB*.xlsx`)がある場合、**新規作成ではなく追記モードで実行する**。DBは使い捨てず、実行のたびに育てる資産である。

1. **正の特定**: 最新の `output/startups_master_YYYYMMDD.jsonl` を「正」とする(JSONLが正、Excelは表示物)。見つからない場合のみ新規作成
2. **重複取得の防止(収集前)**:
   - グローバル: `coresignal_search.py` に `--exclude-jsonl` で**過去の全rawファイル**(output/coresignal_*.jsonl)を渡し、既取得IDのクレジット浪費を防ぐ
   - 国内: 候補抽出エージェントに既存DBの国内社名リストを「除外リスト」として渡す
3. **重複除去(統合時)**: name(小文字)・name_ja・startup_id で既存と突合し、重複は新規側を捨てる
4. **出力**: 既存レコード+新規レコードを結合し、**新しい日付のJSONL/Excel**として保存(古いファイルは上書きせず残す=履歴)
5. **既存レコードの再調査(リフレッシュ)は行わない**。ユーザーが明示指定した場合のみ。ただし `info_date` が6か月超のレコードがあれば完了報告で再調査を提案する

## よくある失敗と防止(エンリッチ・検証で特に注意)

| 失敗 | 正しい形 |
|---|---|
| fitに「高」「良」など独自の値を書く | **○か△の2値のみ**。理由・程度はfit_noteへ |
| mfg_tech_categoryに複数値(";"区切りやリスト)を書く | **1値のみ**。複数該当時は最も中核的な1つ |
| 業界分類を省略形で書く(「モビリティ」等) | 20分類の**語彙をそのまま**(「モビリティ・自動車」) |
| スコア・funding_usdに文字列(「8/10」「約5億円」) | **数値のみ**。原文表記はtotal_funding等の文字列列へ |
| 記憶で書く/それらしい値で埋める | この実行で取得したソースにある事実のみ。不明はnull |
| マッチング用の記述(補完性等)をDBのfit_noteに書く | DBは事実のみ。評価はマッチングDBへ |
| エージェントのエラー終了を見て即再実行 | 先に出力ファイルの存在・件数を確認(書き込み済みが多い) |
| ④シートを手書きのopenpyxlコードで直接編集する | **必ず** build_startup_db.py か update_matching_in_excel.py を使う(A1マーカー=シート特定キーが消えHTML化不能になる) |
| マッチングJSONLのフィールド名を独自に付ける(company_name等) | 「マッチングJSONLの正規スキーマ」の名前をそのまま使う(update_matching_in_excel.py --check-onlyで検証) |

# 能力2: マッチングDB作成

## 起動条件

「マッチングDBを作成して」+ **技術アセット棚卸しのExcelまたはCSV**が提示されること。加えて既存のスタートアップDB(xlsxまたはJSONL)が必要。どちらかが無ければユーザーに確認する。

## 入力: アセット棚卸しファイルの読み方

棚卸しフォーマット: `名称/カテゴリ/成熟度/ステータス/最終更新/技術概要/[重要]スペック|説明/[基本]スペック/[機能]機能グループ>機能|コア・提供価値・制約条件・差別化要素`
- `python scripts/read_assets.py --in <棚卸しファイル> --out output/assets.json` でパースできる。対応フォーマット:
  a) 棚卸し標準形(ヘッダー行「名称/カテゴリ/…/[機能]…」。1シート複数アセット可)
  b) 記入ページ形式(「1.技術概要(技術名/一言説明/主用途/強み)/2.基本スペック/3.機能分解(①②③+提供価値・制約条件・差別化要素)/4.拡張可能性/5.NG条件」。1シート1アセット。「記載例」シートは自動スキップ)
- アセットIDは「名称」を使用(IDカラムがあればそれを優先)

## マッチングDBの項目(キー4列+DB転記4列+本編30列+要約ブロック13列=51列)

社内のアイデア評価DBとフォーマットを揃えるため、列名・列順は同DBの出力形式に合わせる。**スコアはすべて0〜10の数値のみ**(「8/10」等の文字列表記はExcelが日付に化けるため禁止)。

| 区分 | 列 | 内容 |
|---|---|---|
| キー(先頭2列) | スタートアップID/企業名 | スタートアップDBへの結合キー |
| DB転記(4列) | 事業概要/主要プロダクト/プロダクト概要/コア技術 | **③スタートアップDBからstartup_idで自動転記**(build_startup_db.pyが結合)。マッチした相手の実体をシート内で確認するための参照列。エージェントは記入不要 |
| キー(続き2列) | アセットID(名)/協業タイプ | アセットへの結合キー。協業タイプ=技術提供・ライセンス/共同開発/調達・ツール導入/顧客候補/出資 |
| 本編 | No | 連番 |
| 〃 | タイトル | 協業アイデアのタイトル(1行) |
| 〃 | タグ | アセット側+企業側の技術タグ(;区切り) |
| 〃 | 概要 | 協業コンセプト(「アセットの何を、相手の何と組み合わせ、何を生むか」) |
| 〃 | 市場規模(一覧)/CAGR(一覧) | スタートアップDBのmarket_size/market_cagrから転記(協業アイデア固有の市場ならその値) |
| 〃 | 総合スコア(一覧) | =総合評価スコアと同値(一覧表示用) |
| 〃 | 自己評価ランク | **総合評価スコアから自動導出**: A=8.0以上/B=6.0〜7.9/C=6.0未満 |
| 〃 | 想定顧客/課題/解決方法/収益モデル | 協業アイデアとしての4項目 |
| 〃 | アイデアの強み | 協業だからこそ成立する強み |
| 〃 | 使用するコア技術 | 「アセット側: …/スタートアップ側: …」の形式で両方明記 |
| 〃 | スコア8種(各0-10) | 差別化(※補完性を含む: 相手と重複=競合=低スコア)/技術実現性/アセット適合性/業界優位性/業界課題フィット/新規性/ミッション整合性/市場規模スコア/CAGRスコア |
| 〃 | 未使用理由・採用障壁 | アセットの制約条件・協業の障壁 |
| 〃 | コア機能の改善点 | 協業で埋めるべきギャップ |
| 〃 | 市場規模詳細/CAGR詳細 | 出典付き詳細(DB側から転記可) |
| 〃 | 総合評価スコア | スコア8種+市場規模・CAGRスコアの加重平均(既定: 均等。重み指定があれば従う)。0-10・小数1桁 |
| 〃 | エージェントの思考根拠 | スコアの根拠。**事実(両者の技術・実績・出典)のみ引用** |
| 〃 | 主要リスク | 協業アイデアのリスク(評価) |
| 要約ブロック(末尾13列) | 要約_コンセプト/想定市場/顧客/課題/解決方法/市場規模/CAGR/独自性/ミッション整合+スコア4種(独自性/ミッション整合/市場規模/成長性) | 一覧・他システム連携用の凝縮表記(各1文以内・スコアは0-10数値)。本編列の要約であり矛盾させない |

### マッチングJSONLの正規スキーマ(このフィールド名以外を使わない)

ペア生成エージェントには**下記のフィールド名をそのまま**指示に含めること(独自の英語名を発明させない)。No・自己評価ランク・総合スコア(一覧)・DB転記4列はExcel生成時に自動導出されるため**JSONLに書かない**。さらに **`score_total`(9軸平均)と`score_sum_*`4種(本編スコアの転記)は`update_matching_in_excel.py`が自動計算する**ため省略してよい(書いても計算値で上書きされる。モデルに算術をさせない)。

```
startup_id, startup_name, asset_id, coop_type, title, tags, summary,
market_size_list, cagr_list, target_customer, pain, solution, revenue_model,
idea_strength, core_tech_used,
score_differentiation, score_tech_feasibility, score_asset_fit, score_industry_advantage,
score_industry_pain_fit, score_novelty, score_mission_fit, score_market_size, score_cagr,
unused_reason_barriers, core_improvement, market_size_detail, cagr_detail,
score_total, agent_rationale, key_risks,
sum_concept, sum_market, sum_customer, sum_pain, sum_solution, sum_market_size,

```

生成後は `python scripts/update_matching_in_excel.py --in <matching.jsonl> --xlsx <対象xlsx> --check-only` で書き込み前に検証できる(未知フィールド・非数値スコア・要約欠落・③と結合できないIDを警告)。

## マッチング品質の3原則(必須・アイデア発散手法から採用)

協業アイデアの質を上げるため、全ペアで以下3つを守る。単なるタグの重なりで高得点を付けない。

### 原則1: スペック起点で突合する(タグ突合からの格上げ)
「技術キーワードが近い」から入らず、**アセットの最も尖ったスペック・機能(＝強み)を起点**にする。
- 起点の選び方: アセットの[重要]スペック、または[機能]ツリーの「コア・差別化要素」のうち、市場水準より明確に優位な1〜2点(例: 「解析時間8割削減」「接合材レスの無加熱接合」「±0.1mm精度」)
- 問いを反転する: 「このアセットは何に使えるか」ではなく「**この強みが"要求される・直接効く"のはどの企業か**」でDBを探す
- 各ペアの `使用するコア技術`・`概要` に、起点にした強みが相手の何に効くかを必ず書く

### 原則2: 協業価値の足切り判定式 a/b/c(20〜50%改善は採らない)
協業仮説は、少なくとも以下 (a)(b)(c) のどれか1つを満たすものだけを上位・高スコアにする。満たさない「小幅改善」止まりのペアは総合6点未満(Cランク)に抑える。
- **(a) 性能突破**: 相手単独/従来比で主要指標が**桁で変わる**(2倍以上。例: 開発期間半減、精度5倍、寿命3倍)
- **(b) コスト/サイズ/重量が1/10級**: 上位性能は据置で別軸を1桁削減(例: 単価100万→10万、消費電力100W→8W)
- **(c) 別セグメント転用**: アセットの強みを**本来と異なる業界**へ持ち込み、その業界の要求値を満たす(本来業界では既達でも別業界で新規に刺さる)
- どれを満たすかを `エージェントの思考根拠` に必ず明記する(例: 「(a)を満たす: …」)。満たさない場合はその旨と低スコアの理由を書く

### 原則3: 5観点groundingで根拠を厚くする
`エージェントの思考根拠` には、次の5観点のうち**最低2つ**への具体的言及を必ず含める(事実のみ・可能なら出典URL)。
- **主要市場**(規模・成長) / **競合**(既存プレイヤー・差別化) / **顧客課題・ペイン**(具体的な困りごと) / **トレンド**(技術・社会変化・規制の追い風) / **規制**(参入障壁・認証・法規制。例: Euro NCAP、GSR)
- 市場規模・CAGRを新規に使う場合は1ペアにつきWebSearch1回まで、出典が取れた場合のみ記入(取れなければnull)。捏造禁止(事実性の鉄則を全面適用)

## 出力品質基準(良品サンプルの共通特徴・必達)

良品と判定された実出力15ペアの共通特徴を基準化したもの。**全フィールドでこの粒度を満たすこと**。完全な見本2ペアは [references/gold_example.md](references/gold_example.md)(生成前に必ず読む)。

### フィールド別基準(文字数は良品サンプルの実測レンジ)

| フィールド | 文字数 | 必須要素 |
|---|---|---|
| タイトル | 27〜53字 | **協業の成果物・活動を名指す**(例:「自律飛行AI機載コンピュータ向け接合材レス水冷コールドプレート共同開発」)。「X × Y」形式の機械結合は不可 |
| タグ | 5〜6個 | 両者のドメイン語(例: 自律飛行;エッジAI;水冷コールドプレート)。汎用語(接合;熱;金属)の羅列は不可 |
| 概要 | 130〜250字 | ①相手の事業文脈→②アセットのどの強みが直撃するか→③共同で何を作るか、の3段構成。可能ならアセットWANTの逐語引用、双方向価値(相手にとっての価値も)に言及 |
| 課題 | 90〜130字 | **相手側の**切実な課題+「相手が単独では解決困難な理由」(例:「ハードウェア量産経験の少ないAIスタートアップが自力で確立するのは困難」) |
| 解決方法 | 100〜190字 | 実行メカニズム: どの技術がどう働くか+**定量スペックの引用**(誤差5%以内、出力60kW等)+アセットの工法特性がなぜ効くか(例: 流路を型レスで構成できるため最適形状をそのまま製造できる) |
| アイデアの強み | 60〜120字 | 技術補完性に加えて**外部事実**(出資・資本関係、既存顧客、国内同士で契約しやすい等)。「相乗効果」だけの記述は不可 |
| 使用するコア技術 | 85〜125字 | 「アセット側: …/スタートアップ側: …」形式で両方に具体技術名 |
| 採用障壁 | 90〜170字 | 実在する制約: 技術開示制約(NDA前提)・工法制約(加圧に有利な形状等)・相手のステージ(PoC段階なら先行投資リスク) |
| コア機能の改善点 | 50〜120字 | 埋めるべき技術ギャップを具体的に(精度の未検証領域、専用テンプレ整備、制約条件の組み込み等) |
| エージェントの思考根拠 | 170〜290字 | **軸ごとの採点理由を点数付きで**(例:「asset_fitはWANT第2項に直撃のため9」「PoC段階のため6に抑えた正直採点」)。**低い軸ほど理由を書く**。市場規模・CAGRの出典確認方法とscore_totalの算出法も明記 |
| 主要リスク | 80〜170字 | **①②③の番号列挙で3〜4件**(相手ステージの流動性・技術の精度移転未検証・競合の追随・市場解釈の注意点等) |

### 全体特性(行間・スコアの基準)

- **スコアは0〜10の整数**で行ごとに実質的な差をつける。良品サンプルの分布は3〜9(低スコアを恐れない。総合4.9のCランクも収載されている)。**全行B・等差列・同一パターンは不合格**
- **正直採点**: 弱点のある軸は下げ、その理由を思考根拠に書く(「〜のため6に抑えた」)。全軸7前後に寄せる中庸採点は不可
- **協業タイプは使い分ける**(良品サンプルは顧客候補9/共同開発5/調達・ツール導入1)。全行「共同開発」は突合が浅い兆候
- **同一文の行間コピペ禁止**: 課題・解決方法・強み・根拠は行ごとに固有の内容(企業名を差し替えただけの文は不合格)
- 市場規模詳細・CAGR詳細は「値+出典名+(WebSearchで出典確認)」の形式。複数ソースがあれば併記

## 能力2の手順

1. アセットファイルをパース(`read_assets.py`)し、各アセットの[機能]ツリー(コア・提供価値・制約条件・差別化要素)+[重要]スペックをタグ化。**この時点で各アセットの「最も尖った強み1〜2点」を特定しておく**(原則1の起点)
2. スタートアップDBを読み、突合: ①業界分類・製造技術分類で粗く絞る→②**原則1のスペック起点**でアセットの強み×企業のtech_keywords・core_tech_description・product_summaryを突合(「この強みが効く相手」を探す)→③有望ペアに協業タイプを付与
3. ペアごとに協業アイデアを生成し本編+要約の43データ列を記入(キー・転記列・No・ランク等は自動。要約13列は本編の凝縮)。**原則2(a/b/c判定)と原則3(5観点grounding)を各ペアで満たす**。スコアは0-10数値、根拠は事実のみ。件数はユーザー指定、未指定なら**アセット1件あたり上位5〜10ペア**を提案
   - **生成エージェントには [references/gold_example.md](references/gold_example.md) を必ず読ませ、「この2ペアと同じ粒度・構成で書く」ことと「出力品質基準」(上記)の文字数レンジ・必須要素を指示に含める**。禁止語:「相乗効果」「技術融合」「唯一の実現」「(単独の)〜による課題解決」「事業展開」— これらが出たら書き直し
4. Excelへの書き込みは**必ずスクリプトで行う**(手書きのopenpyxlコードで④シートを直接編集することを禁ずる。A1マーカー「マッチングDB(N件)」が消える・列名がズレる・要約13列が欠落し、能力3のHTML化が失敗する):
   - スタートアップDBがJSONLである場合: `python scripts/build_startup_db.py --in <DBのjsonl> --matching <matching jsonl> --assets output/assets.json --out <xlsx>` で**④マッチングDB**(DB転記4列は自動結合)と**⑤アセット情報**(使用アセットの全情報を転記保存)を生成
   - **既存のマッチング用xlsx(③⑤入り)しか無い場合**: `python scripts/update_matching_in_excel.py --in <matching jsonl> --xlsx <既存xlsx>` で④シートだけを正規フォーマットで再生成(③からの転記4列・No・ランクは自動。まず`--check-only`で検証してから書き込むこと)
5. 完了報告: ペア数/ランク分布/使用アセット数/**各ペアが満たした判定式(a/b/c)の分布**/セルフチェック(3〜5行の根拠突合)結果
6. 検証で**[品質]警告**が出た場合は「品質改善パス」(下記・最大3回)をユーザーに提案する

## 能力2の品質改善パス(再生成・要約充足) — 最大3回まで

④生成後の検証(`update_matching_in_excel.py --check-only`)で**[品質]警告**が出た場合、またはユーザーが再生成・要約充足を指示した場合に実行する。検証は「出力品質基準」を内容レベルで機械チェックする: a/b/c判定なし/概要が定型文/**禁止語**(相乗効果・技術融合等)/**文字数が基準レンジ未満**/**タイトルが「X × Y」機械結合**/**主要リスクが①②列挙でない**/**スコアが非整数・等差列・全行同一ランク**/**行間コピペ**(企業名差し替えの定型量産)/**英語スラッグ漏れ**/要約13列欠落/**定型プレースホルダー残り**(「詳細分析」「予定」等)。

**初回生成もパス1と数え、パス内で完結させる**: 生成 → `--check-only` → 警告の対象行をその場で修正 → 再検証 → 書き込み。「書いてから次のパスで直す」を繰り返すと3回で収束しない。

**採用障壁(`unused_reason_barriers`)・改善点(`core_improvement`)・主要リスク(`key_risks`)は必ず中身を書く**。「詳細分析が必要」「後で検討」等の仮テキストで埋めてはならない(検証で[品質]警告になる)。事実が不明で書けないなら**空欄**にする(空欄は正、仮テキストは不正)。市場規模詳細/CAGR詳細は出典が取れた場合のみ記入し、取れなければ空欄。

- **モードA: 要約充足** — 本編列は変更せず、空の要約13列(`sum_*`/`score_sum_*`)だけを本編の内容から凝縮して追記。新規調査・WebSearch不要
- **モードB: 再生成** — [品質]警告の対象行だけを3原則準拠で再生成(原則2のa/b/c判定を思考根拠に明記、原則3の5観点grounding、行ごとの実質的なスコア評価)。1エージェント最大8ペアで分割起動。事実性の鉄則を全面適用

**手順(1パス):**
1. `--check-only` で[品質]警告と対象行を特定
2. **run_log.md に「品質改善パス N/3」を記録してから実行する**(Nは必ず数える。記録なしで実行しない)
3. JSONLは上書きせず新しい日付のファイルに保存 → `update_matching_in_excel.py` で④を再生成 → 再検証
4. [品質]警告が0になったらその時点で終了(3回使い切る必要はない)

**停止条件(いずれかに達したら停止し、ユーザーに報告して指示を仰ぐ):**
- [品質]警告が0になった(完了)
- **3パス実行済み**(残った警告と解消できなかった理由を報告。4回目以降を自走しない)
- **同一の警告が2パス連続で解消しない**(手法を変えても直らないものはユーザー判断に委ねる)

# 能力3: マッチング結果HTMLレポート生成

マッチングExcel(④マッチングDB/⑤アセット情報/③スタートアップDB)を、閲覧・共有用の**1ファイル完結の静的HTML**に変換する。トリガー: 「マッチング結果をHTMLにして」「レポートにして」等、またはマッチングDB生成後の提案。

```
# 最小: 入力Excelだけ渡せば、同じ場所に「マッチング結果_YYYYMMDD.html」を自動生成
python scripts/build_matching_html.py --in <マッチング.xlsx>
# 引数なし: cwd と ./output から④マッチングDBを持つ最新xlsxを自動検出
python scripts/build_matching_html.py
# 明示指定
python scripts/build_matching_html.py --in <xlsx> --out <html> [--title "..."] [--subtitle "..."]
```

- **`--out`省略で入力と同じ場所に日付付き自動命名／`--in`省略で近傍(cwd・output/)のマッチングExcelを自動検出**するため、DB生成直後は引数なしで即HTML化できる。
- シートは**先頭セルの内容**で特定(①〜⑤の採番差異に非依存)、列は**ヘッダー名**で引く(列順差異に耐性)。CSS/JSは `scripts/matching_html_style.css` / `matching_html_app.js` を読み込む(見た目の調整はこの2ファイルを編集)。
- 生成される構造(この構造を標準とする):
  - 上部: **対象技術アセット一覧**(⑤から。名称+成熟度+概要+WANT。**クリックで絞り込み**) → フィルタバー(総合スコア降順)
  - カード: **「技術アセット列 | スタートアップ列」の2段組**(下端に「本協業で使う技術」を揃える)。アセット列=⑤の概要+主要スペック、スタートアップ列=④のDB転記4列+③のWebサイト
  - **協業アイデア**(強調ボックス): タグ/課題/解決策/アイデアの強み/市場規模のみ常時表示
  - **折りたたみ**: 詳細スコア(9軸バー)・想定顧客・収益モデル・評価根拠・採用障壁・改善点・主要リスク・市場規模/CAGR詳細
- 帯色=協業タイプ、バッジ枠/バー色=ランク(A緑/B紺/C橙)。9軸=アセット適合性/技術実現性/差別化/業界優位性/業界課題フィット/新規性/ミッション整合性/市場規模/成長性。
- 公式サイトURLは④に無いため③スタートアップDB(startup_idで結合)から取得。③が無い場合はURL行を省略。
