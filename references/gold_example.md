# マッチングDB 良品見本(gold example) — この粒度を再現する

実出力の良品から「書き方の型・粒度」を抽出し、**企業・技術・数値・出典をすべて架空に置き換えた見本**。ペア生成エージェントへの指示に必ずこのファイルを読ませ、「この2ペアと同じ粒度・構成で書く」ことを要求する。フィールド名は正規JSONLスキーマ。

> **警告(事実性の鉄則)**: 以下の企業名・技術名・数値・市場データ・出典名はすべて架空であり、実在の企業・調査とは無関係。**見本から学ぶのは書き方の型と粒度のみ。値・社名・出典を実出力に流用したら捏造になる。**

見本の型が異なる点に注意: 見本1=共同開発型(アセット側が部品を供給)、見本2=調達・ツール導入型(逆方向=アセット側が顧客)。**協業タイプによって書くべき内容が変わる**ことも見本から学ぶこと。

- 架空アセット: 「無加熱拡散接合による水冷コールドプレート」(強み: 接合材レスでクラック・腐食リスクなし/流路レイアウト自由/ハイサイクル量産。WANT: 「高発熱AIプロセッサ向け冷却部品の用途開拓」)
- 架空スタートアップ1: エアロブレイン(物流ドローン向け自律飛行AIと機載コンピュータ。PoC段階・モビリティ部品大手が出資)
- 架空スタートアップ2: シミュフロー(機械学習サロゲートによる熱流体解析SaaS「FlowSurro」。従業員20名規模)

## 見本から学ぶべき「書き方の型」(全ペア共通)

1. **タイトル** = 「(相手の文脈)向け(具体的成果物)(協業形態)」。成果物名詞で終わる
2. **概要** = 相手の事業文脈(1文) → アセットのどの強みが直撃するか(1文) → 共同で何をするか(1文) → 可能ならWANT引用・双方向価値(1文)
3. **課題** = 相手側の技術的・事業的ペイン + 「相手が単独では困難な理由」を必ず含める
4. **解決方法** = どの技術がどう働くか(定量スペック逐語引用) + アセット工法特性が効く理由 + 量産・スケールへの道筋
5. **アイデアの強み** = 補完性の構造(双方向なら両方) + 外部事実(出資関係・既存顧客・地理・契約のしやすさ)
6. **採用障壁** = アセットの開示制約/工法制約 + 相手のステージに起因する現実的な障壁
7. **改善点** = 「未検証の領域」「整備すべきテンプレ・プロセス」「組み込むべき制約条件」を名指す
8. **思考根拠** = 軸名+点数+理由を高低両方(特に**下げた軸の理由**)。市場データの出所(WebSearch確認 or DB記載値)とscore_total算出法を明記
9. **主要リスク** = ①②③④で3〜4件。相手ステージ・精度移転・競合追随・スコア解釈の注意
10. **スコア** = 0〜10整数・行ごとに実差。強い軸9、弱い軸4〜6の正直採点。総合が6未満ならCランクでよい(無理にBに乗せない)
